# 从一个好的项目目录开始
一个项目的目录结构是整个项目推进的地基，设计好的项目目录会直接决定我们如何使用、更新、管理整个实验。

一种可靠的目录结构设计
```
my-project/
├── README.md
├── pyproject.toml          # 依赖 + 把 src 装成可导入的包
├── .gitignore
├── .venv                   # 0 虚拟环境管理，可选uv或者conda
├── configs/                # 1 所有配置,定义"实验是什么"
│   ├── config.yaml         #    主入口配置
│   ├── model/              #    可组合的配置组
│   │   ├── resnet50.yaml 
│   │   ├── vit_base.yaml 
│   │   └── my_method.yaml  
│   ├── dataset/
│   ├── optim/
│   └── experiment/         #    成套的实验预设
│
├── src/myproject/          # 2 库代码,所有真正的逻辑
│   ├── __init__.py
│   ├── data/               #    数据集、加载、预处理
│   ├── models/             #    模型定义
│   ├── losses/
│   ├── trainer/            #    训练循环
│   ├── eval/               #    指标、评估协议
│   └── utils/              #    种子、日志、git 记录等基础设施
│
├── scripts/                # 3 入口脚本,薄薄一层
│   ├── train.py
│   ├── eval.py
│   └── summarize.py        
│
├── data/                   # 4 数据,不进 git
│   ├── raw/                #    原始数据,只读、永不修改，可以使用链接
│   ├── processed/          #    预处理产物
│   └── README.md           #    数据来源、版本、下载方式
│
├── runs/                   # 5 实验产出,不进 git
│   └── 2026-06-05_14-30_baseline_seed0/
│
├── notebooks/              # 探索性分析,不放主流程逻辑
└── tests/                  # 单元测试
```
整个目录的设计上遵循几点原则：
1. 整个实验的所有文件大体上有不同的类别，它们的**修改需求、读取需求、使用方式、信息容量各不相同**，所以使用不同级别、不同名称的目录进行区分是十分有必要的
2. 一个项目的整个生命流程需要经过多次的更新和变动，**对项目的变动进行准确的版本保存和备注记录是不可或缺的**
3. 一次版本的实验结果是配置+代码+数据+随机性组合得到的，因此从起点来看，**配置、代码、数据的组合管理十分重要**，而从结果来看，结果的可复现性以及信息准确性也需求与配置、代码、数据进行严格的对应

而可靠的目录设计背后的思考是：
**代码、配置、数据、产出各自隔离**，它们的改动频率、版本控制需求、文件体积、读写需求不同。所以我们因此建立了四个层级的一级目录：
	1. src/ :核心代码，这里是我们对整个模型结构、流程进行变动的关键内容，在开发阶段会不断进行更新和迭代，这里是版本控制的关键区域
	2. config/ :核心的配置，实验内部的参数统一使用代码外的配置文件来进行管理，需要与实验进行对应
	3. script/:批处理脚本，对src里的运行内容进行封装，主要完成批处理的实验进行
	4. data/:数据，体积大且基本只读
	5. runs/:结果产出，对应实验结果的记录和保存

# 配置管理
最简单的配置管理就是所有关键参数写在代码文件中，每一次要跑新的实验就需要改动一次代码再运行。这种方式麻烦、且各种配置项很容易变得分散，改动不方便

再之后就是使用一个单独的yaml文件统一记录参数，然后在代码中读取这个yaml文件中的内容，使用层级递进的方式来获取配置项的值。这样的话分散以及不好修改的问题解决了，但是多种配置版本的保存和隔离又出现了问题

当前的一种最优解是按照组来进行切分，搭配Hydra这种库进行管理

比如，一个配置可能包括模型上的、数据集读取上的、优化器上的等等，我们把这种“正交”的维度单独区分成组，**最终的实验配置由这些组拼接得到**
```yaml
defaults:
  - model: resnet50
  - dataset: cifar100
  - optim: adamw
  - trainer: default
  - _self_

seed: 0
output_dir: runs/
```

这样，我们如果想要尝试不同的模型、不同的数据集、不同的优化器，就可以只单独替换组里的内容了。在对比实验中，我们可以直接替换掉model里的内容，在消融实验中，我们控制指定模型中的开关来实现
```yaml
# experiment/ablation_no_moduleA.yaml
defaults:
  - /experiment/ours_full   # 继承完整方法
  - _self_

model:
  use_module_A: false       # 只改这一个开关
```

# 运行结果保存
runs是实验的档案馆，用于封存实验真正跑出来的所有信息

按照一般的可行标准，一个比较标准的run目录里应该有下面这些内容：
```
runs/2026-06-05_14-30_ours-full_seed0/
├── config.yaml          # 1 解析后的完整配置(不是模板,是最终值)
├── git_info.txt         # 2 代码版本:commit hash + 是否 dirty + diff
├── env.txt              # 3 环境:Python/库版本、GPU、CUDA
├── metrics.csv          # 4 结构化指标:每个 step/epoch 一行
├── log.txt              # 5 完整运行日志(stdout/stderr)
├── summary.json         # 6 最终关键指标,一个文件几个数,给汇总脚本读
├── checkpoints/         # 7 模型权重
│   ├── best.ckpt
│   └── last.ckpt
└── status              # 8 状态标记:RUNNING / DONE / FAILED
```
- config.yaml:首先需要的就是一个完整了配置文件，这个存储了一次实验的完整配置信息。config.yaml内的内容应该是解析之后的完整版本
- checkpoints/:另一个必须要保存的内容是checkpoint，保存了关键的结果内容。这里需要注意的是**保存的频率和总数**（可以使用滚动刷新）
- git_info.txt:其次需要保存的是git信息，这里代表的是对应的代码
- log.txt:有必要保存程序整体运行的完整日志，记录实验运行时的警告和报错信息
- env.txt:环境信息是一个可以记录的内容，虽然对实验的影响不大
- status:记录整个实验的状态，便于观察的话可以直接用文件名来保存这类信息
- metrics.csv：记录实验中所有的指标情况，使用一个单独的表格文件来整理会更加清晰

runs/文件夹下的一个关键是整个目录的命名

所有的实验都需要保存在runs目录下，但是单个实验的目录如何互不影响且能信息清晰也是需要注意的。一般用 时期+时间+语义描述+关键特征来表达

# 代码运行侧的联动
代码最后、也是最关键的封装就是train.py这一个文件

我们需要在这个文件中组合我们的模型组件模块、确定配置输出、结果保存、日志记录的方式

大体上可以按照四个阶段来搭建：
1. 阶段零：在main函数的部分之前增加Hydra的装饰器，用于导入预先准备好的config文件（这里按组来组织）
2. 阶段一：在所有模型构建和训练循环之前先进行实验的环境记录封存。主要需要包括的是在runs下建立带时间戳的目录、输出完整的config内容、输出完整的环境内容、输出日志到文件、保存git信息、固定随机源
3. 阶段二：构建模型，包括数据集对象、模型对象、优化器对象、记录器（wandb、swanlab、tensorboard）
4. 阶段三：完整的训练循环，可以使用异常来兜底，同步到status文件中去
